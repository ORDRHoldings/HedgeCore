§01

Platform Overview

Hedge Terminal is an institutional-grade FX risk management infrastructure platform designed for treasury operations, risk committees, and portfolio managers. It provides deterministic hedge construction, multi-scenario stress analysis, and audit-ready committee documentation within a single integrated workflow.

Design Philosophy
  • Deterministic computation — no ML black boxes; every output is fully traceable
  • Committee-grade documentation at every step of the workflow
  • IFRS 9, ASC 815, and IAS 39 compliance awareness built into output structure
  • Zero-touch export for external audit systems (PDF, Excel, Audit JSON)

§03

CurrencyFX Module

What It Does
  • Ingests accounts payable (AP) and accounts receivable (AR) trade records in foreign currency
  • Nets AR against AP by currency and tenor (calendar month buckets)
  • Applies configurable hedge ratios to confirmed and forecast positions
  • Accounts for existing hedge instruments on the books (NDF, FWD)
  • Prices new hedge instruments using a live or demo forward curve
  • Produces a deterministic, snapshot-bound Committee Pack with full SHA-256 cryptographic attestation
5-Step Invoice-Driven Workflow
01
Commercial Exposure
Load or manually enter AP/AR trade positions by currency, entity, amount, and tenor. Supports CSV upload or demo fixture selection. Required gate before proceeding.
02
Market Conditions AUTO
Auto-populates spot rate and forward curve from invoice currency and due date. No user action required — triggers automatically on step entry when exposure data is present.
03
Hedge Policy
Select a policy preset governing confirmed/forecast hedge ratios, execution product (NDF/FWD), and minimum trade size. Five presets available plus fully custom configuration.
04
Risk Mitigation AUTO
Auto-fills hedge recommendations derived from Exposure + Market + Policy + Hedge Intent. Existing hedge overlay applied. Residual gap calculated per calendar month bucket.
05
Authorization
Gate check runs all 21 validation codes. Displays integrity score (0–100) and validation status. Generate Hedge Plan sends inputs to deterministic kernel and redirects to Committee Pack.

§04

Computation Engine

13-Step Deterministic Kernel
  1. Input deserialization and normalization
  2. Currency pair detection and base currency determination
  3. Temporal bucketing (calendar month aggregation)
  4. Net exposure calculation per bucket (AP minus AR)
  5. Existing hedge overlay and netting
  6. Gap analysis — uncovered exposure per bucket
  7. Policy application (confirmed/forecast ratio enforcement)
  8. Forward rate interpolation from curve
  9. New hedge ticket generation (notional, rate, dates)
  10. Cost estimation (spread × notional)
  11. Scenario stress testing (4 scenarios: base, adverse, severe, extreme)
  12. Validation (21 codes: V-001 through V-021)
  13. Run envelope generation (SHA-256 hash, timestamp, engine version)
Validation System — 21 Codes (V-001 through V-021)
CategoryScopeOn Failure
Input CompletenessMissing fields, invalid dates, malformed identifiersFAIL
Policy ComplianceRatio breaches, minimum trade violations, product mismatchesFAIL
Market Data QualityStale data, missing forward points, curve gapsWARNING
Hedge IntegrityExpired hedges, direction mismatches, notional overflowsFAIL

Status definitions: PASS all checks clear    WARNING non-blocking    FAIL blocks computation

Snapshot Locking & Determinism

All inputs are locked at run time into a cryptographic snapshot. Any identical input set produces byte-identical outputs. SHA-256 hashes are computed over 6 datasets: INPUTS, OUTPUTS, TRADES, HEDGES, MARKET, and POLICY.

Determinism Guarantee
Engine v1.0.0 guarantees that given the same input snapshot, the computation kernel will produce identical outputs on every run, on every machine, at any point in time. There is no randomness, no time-dependent sampling, and no external data fetching during computation.

§06

Pipeline Architecture

Tri-State Governance Model
SANDBOX
Active calculation workspace. All input and computation. Fully mutable.
→
STAGING
Pending approval queue. Outputs submitted for review. Immutable once submitted.
→
LEDGER
Archived record. Approved runs stored permanently. Cryptographically sealed.

The pipeline enforces a linear, one-way governance flow. Runs cannot be moved backward. Each stage transition requires explicit user action and produces a new cryptographic attestation record.

Trace Bundle — 6 Computation Stages
PARSE
→
VALIDATE
→
NORMALIZE
→
KERNEL
→
SCENARIO
→
AUDIT

Each stage produces a SHA-256 hash of its inputs and outputs, forming an unbroken chain. The final AUDIT hash signs the entire computation. Any tampered intermediate result breaks the chain at the point of modification.

§07

Security & Compliance

Cryptographic Integrity
  • SHA-256 cryptographic attestation on all computation inputs and outputs
  • Snapshot-bound determinism — byte-identical reproduction guaranteed from identical inputs
  • 6-stage trace bundle with unbroken hash chain (PARSE → VALIDATE → NORMALIZE → KERNEL → SCENARIO → AUDIT)
  • No client-side randomness in any computation path
Regulatory Awareness
  • IFRS 9 — Hedge effectiveness testing structure built into output framework
  • ASC 815 — US GAAP derivative and hedge accounting disclosure support
  • IAS 39 — Historical standard awareness for legacy portfolio documentation
Access Control
RoleCapabilities

risk_officerView results, approve runs, promote runs to Staging
committee_memberView Committee Pack, sign off on committee reports

Infrastructure
  • All computation executed server-side on Render infrastructure — no sensitive data processed in browser
  • Cookie-backed session authentication with server-side validation on every request
  • CORS enforced on backend API with strict origin allowlist

Glossary

TermDefinition
APAccounts Payable — the entity owes FX to a counterparty (creates long foreign currency exposure)
ARAccounts Receivable — the entity is owed FX by a counterparty (creates short foreign currency exposure)
NDFNon-Deliverable Forward — cash-settled FX forward; no physical exchange of currency at maturity
FWDDeliverable Forward — physically settled FX forward; full notional exchange occurs at maturity
Coverage RatioProportion of net exposure covered by existing and new hedge instruments (as a percentage)
Hedge RatioTarget coverage percentage applied per position type (CONFIRMED or FORECAST) as defined in policy
BucketCalendar month aggregation period used to group exposures and hedges by settlement tenor
SnapshotPoint-in-time lock of all inputs (trades, hedges, market, policy) producing a deterministic computation
Run IDUUID uniquely identifying a single computation run — present on all output artifacts for traceability
Integrity Score0–100 score computed from an 8-rule validation waterfall; 100/100 = fully clean inputs
Committee PackFull output bundle: Committee Summary, Exposure & Buckets, Scenario Analysis, Trade Tickets, Audit Evidence, and R-01 through R-06
Gate CheckPre-computation validation evaluating all 21 codes — blocks computation on any FAIL result
Forward PointsIncremental rate adjustment by tenor applied to the spot rate to derive the forward exchange rate per bucket
TenorTime period to settlement (M1 = 1 month, M2 = 2 months, etc.)
PolicyConfiguration governing hedge ratios (confirmed/forecast), execution product (NDF/FWD), and minimum trade size
SHA-256Cryptographic hash algorithm (256-bit) used for input attestation and trace bundle integrity verification
Trace BundleOrdered chain of 6 SHA-256 hashes (PARSE → VALIDATE → NORMALIZE → KERNEL → SCENARIO → AUDIT)
CONFIRMEDPosition status indicating high certainty of settlement — typically an invoice already received
FORECASTPosition status indicating anticipated but not yet confirmed settlement — lower hedge ratio applies
Residual ExposureUncovered FX exposure remaining after netting AR against AP and overlaying existing hedge instruments
